# PRD: Poker Settlement Telegram Bot

**Версия:** 1.0
**Дата:** 10 апреля 2026
**Статус:** Draft — готов к разработке
**Автор:** Product Manager (совместно с основателем проекта)

---

## 1. Обзор приложения и цели

### 1.1 Проблема
Во время дружеской игры в покер участники многократно докупаются фишками, теряют счёт потраченным суммам и в конце вечера тратят значительное время на ручной подсчёт того, кто кому и сколько должен. Процесс подвержен ошибкам и конфликтам.

### 1.2 Решение
Telegram-бот, который ведёт учёт бай-инов, докупов и финальных остатков фишек для каждой игры, автоматически рассчитывает минимальный набор переводов между участниками и рассылает каждому персональные реквизиты для расчёта.

### 1.3 Цели проекта
- **Главная цель:** полностью автоматизировать финансовые расчёты после покерной игры в постоянной компании друзей.
- **Второстепенные цели:**
  - Обеспечить прозрачность (все видят счётчики докупов в реальном времени).
  - Минимизировать количество переводов через оптимизационный алгоритм.
  - Сохранить историю игр для возможной будущей аналитики.

### 1.4 Не-цели (Non-goals)
- Бот **не является** публичным продуктом для произвольных компаний. Он создаётся для одной постоянной группы друзей.
- Бот **не обрабатывает** реальные платежи — только рассчитывает долги и показывает реквизиты.
- Бот **не участвует** в игровом процессе покера (не раздаёт карты, не считает банк на столе).

---

## 2. Целевая аудитория

**Основные пользователи:** постоянная группа друзей (~4–10 человек), играющих в покер офлайн на живые деньги.

**Характеристики:**
- Пользуются Telegram ежедневно.
- Имеют счета в российских банках, поддерживающих СБП.
- Доверяют друг другу (нет риска мошенничества со стороны участников).
- Технически базовые пользователи — от бота ожидается максимально простой UX.

## 3. Основные функции и функциональность

### 3.1 Онбординг нового игрока

**Описание:** при первом запуске бот собирает контактные данные игрока, необходимые для формирования реквизитов перевода.

**Флоу:**
1. Игрок отправляет `/start` в личный чат с ботом.
2. Бот приветствует игрока и просит поделиться номером телефона.
3. Игрок выбирает один из двух вариантов:
   - Нажимает кнопку **«Поделиться контактом»** (Telegram `KeyboardButton` с `request_contact=true`) — номер передаётся автоматически.
   - Нажимает **«Ввести номер вручную»** — вводит номер в международном формате текстом.
4. Бот валидирует формат номера и переходит к выбору банка.
5. Бот показывает inline-клавиатуру со списком популярных банков: `Тинькофф`, `Сбербанк`, `Альфа-Банк`, `ВТБ`, `Райффайзен`, `Озон Банк`, `Яндекс Банк`, `Другой`.
6. При выборе `Другой` бот просит ввести название банка текстом.
7. Бот подтверждает сохранение профиля и показывает финальное сообщение с данными (имя из Telegram-профиля, телефон, банк).

**Acceptance criteria:**
- Игрок не может создавать или присоединяться к играм до завершения онбординга.
- Номер телефона сохраняется в международном формате `+7XXXXXXXXXX`.
- Повторный запуск `/start` для уже зарегистрированного игрока показывает его текущие данные и кнопку «Ок» (не перезатирает профиль).
- Имя игрока по умолчанию берётся из `first_name` + `last_name` Telegram-профиля.

### 3.2 Управление профилем (отложено во Future Enhancements)
В MVP изменение номера телефона и банка **не предусмотрено** через интерфейс бота. При необходимости корректировки — ручное изменение в БД или переонбординг.

**В MVP доступно только изменение отображаемого имени:**
- Команда `/name <новое имя>` — игрок задаёт произвольное отображаемое имя (полезно, когда в компании несколько Саш).
- По умолчанию имя = `first_name` + `last_name` из Telegram.

### 3.3 Создание игры

**Описание:** любой зарегистрированный игрок может создать новую игровую сессию.

**Флоу:**
1. Игрок отправляет команду `/newgame` в **личный чат** с ботом или в **общий групповой чат** (бот должен быть добавлен в группу).
2. Бот спрашивает размер бай-ина с подсказкой: «Укажи бай-ин в рублях (по умолчанию 1000)».
3. Игрок вводит число или нажимает кнопку **«1000 ₽ (по умолчанию)»**.
4. Бот создаёт игру и публикует в **общем групповом чате** сообщение-хаб игры (см. раздел 3.4).
5. Создатель автоматически становится первым участником игры.

**Acceptance criteria:**
- Бай-ин должен быть положительным целым числом от 100 до 100 000 ₽.
- В один момент времени в групповом чате может быть только **одна активная игра**. При попытке создать вторую бот отвечает: «В чате уже идёт игра #N. Заверши её перед созданием новой».
- Создатель игры фиксируется и отображается в сообщении-хабе.
- Соотношение фишек к рублям фиксировано: **1 фишка = 1 рубль**.

### 3.4 Сообщение-хаб игры в общем чате

**Описание:** центральный элемент UI — обновляемое в реальном времени сообщение в групповом чате, показывающее состояние игры.

**Содержимое сообщения:**
```
🎰 Игра #42 — активна
Бай-ин: 1000 ₽
Создатель: Вася

Игроки (4):
• Вася — 2 докупа (3000 ₽)
• Петя — 0 докупов (1000 ₽)
• Коля — 3 докупа (4000 ₽)
• Саша — 1 докуп (2000 ₽)

💰 Банк: 10 000 ₽
⏱ Начало: 19:45
```

**Inline-кнопки под сообщением:**
- `➕ Присоединиться` — добавляет игрока в игру.
- `💵 Докупиться` — фиксирует докуп для игрока, нажавшего кнопку.
- `➖ Отменить докуп` — уменьшает счётчик докупов игрока на 1 (если > 0).
- `🏁 Завершить игру` — запускает процесс финального расчёта (с подтверждением).

**Acceptance criteria:**
- Сообщение обновляется через `editMessageText` после каждого значимого действия.
- Кнопки `Докупиться` и `Отменить докуп` действуют **только на того, кто их нажал** (бот использует `callback_query.from.id`).
- Присоединиться можно в любой момент, пока игра активна — даже после начала.
- Если игрок, не зарегистрированный у бота, нажимает «Присоединиться», бот отвечает ему приватным сообщением (через `answerCallbackQuery` с `show_alert=true`): «Сначала нажми /start в личном чате со мной, чтобы зарегистрироваться».
- `Отменить докуп` не может сделать счётчик отрицательным.
- Те же действия (присоединение, докуп, отмена докупа) доступны дублирующими inline-кнопками в **личном чате с ботом** через команду `/game` — игрок получает персональное сообщение с теми же кнопками.

### 3.5 Завершение игры и сбор результатов

**Описание:** процесс перехода от активной игры к финальному расчёту долгов.

**Флоу:**
1. Любой участник нажимает кнопку `🏁 Завершить игру` в сообщении-хабе.
2. Бот показывает **подтверждение** через `callback_query` alert: «Точно завершить игру? Это действие запустит сбор результатов у всех игроков».
3. При подтверждении игра переходит в статус `collecting_results`.
4. Бот **редактирует сообщение-хаб** в общем чате, заменяя его на сообщение сбора результатов:
   ```
   🏁 Игра #42 — сбор результатов
   Банк: 10 000 ₽

   Ожидаем данные от игроков:
   ⏳ Вася
   ⏳ Петя
   ⏳ Коля
   ⏳ Саша
   ```
5. Бот отправляет **персональное сообщение** каждому участнику в личный чат:
   ```
   🏁 Игра #42 завершена, пора подвести итоги.

   Твои данные:
   Бай-ин: 1000 ₽
   Докупов: 2  [➖] [➕]
   Всего вложено: 3000 ₽

   Сколько у тебя осталось?
   [ Ввести в фишках ] [ Ввести в рублях ]
   ```
6. После нажатия на одну из кнопок ввода бот переходит в режим ожидания числа от этого игрока.
7. Игрок отправляет число. Бот показывает подтверждение:
   ```
   Проверь данные:
   Докупов: 2
   Осталось: 2000 ₽
   Результат: −1000 ₽

   [✅ Подтвердить] [✏️ Изменить]
   ```
8. После подтверждения игрок отмечается как `✅` в сообщении в общем чате.
9. Игрок может вернуться и изменить данные, пока **хотя бы один** другой игрок ещё не подтвердил свои результаты. После финального расчёта изменения невозможны.

**Acceptance criteria:**
- Подтверждение завершения игры обязательно — случайный клик не должен завершать игру.
- Персональное сообщение отправляется только зарегистрированным участникам игры.
- Счётчик докупов в финальном сообщении можно скорректировать кнопками `➖` и `➕` (но не ниже нуля).
- Игрок может выбрать ввод в фишках или в рублях. Так как соотношение 1:1, валидация одинаковая: целое число ≥ 0.
- Пока не все игроки подтвердили результаты, любой из них может вернуться в меню и пересчитать через команду `/edit` в личке.
- Сообщение-хаб в общем чате обновляется после каждого нового подтверждения (через `editMessageText`).

### 3.6 Валидация и расчёт долгов

**Описание:** после того как все игроки подтвердили свои данные, бот проверяет сходимость банка и, если всё корректно, рассчитывает оптимальные переводы.

**Флоу:**
1. Бот вычисляет ожидаемый банк: `Σ(бай-ин + докупы_игрока × бай-ин)`.
2. Бот вычисляет фактический банк: `Σ(остаток_игрока)`.
3. **Если суммы не сходятся:**
   - Бот отправляет в общий чат сообщение:
     ```
     ⚠️ Расхождение в подсчёте!
     Ожидаемый банк: 10 000 ₽
     Сумма остатков: 9 200 ₽
     Разница: −800 ₽

     Проверьте данные. Кто-то ошибся с количеством докупов или остатком.
     Используйте /edit в личке, чтобы исправить.
     ```
   - Игра остаётся в статусе `collecting_results`, игроки могут править данные.
4. **Если суммы сходятся:**
   - Бот запускает алгоритм минимизации транзакций (см. раздел 7.1).
   - Бот рассылает **персональные сообщения** каждому участнику с результатом и инструкциями к переводу (или получению).
   - Бот публикует **итоговую сводку** в общем чате.
   - Игра переходит в статус `finished`.

**Acceptance criteria:**
- Финальный расчёт запускается **только** когда все игроки нажали «Подтвердить».
- При расхождении банка расчёт **не производится** и данные не сохраняются как финальные.
- Точность расчётов — до 1 рубля (целые числа, без копеек).

### 3.7 Персональное сообщение с результатом

**Для проигравших:**
```
🎰 Игра #42 завершена!

Твой результат: −1000 ₽ 📉
Бай-ин: 1000 ₽, докупов: 2 (+2000 ₽)
Итого вложено: 3000 ₽
Осталось фишек: 2000 ₽

💸 Тебе нужно перевести:
• Коле — 800 ₽
  Тинькофф: +7 999 123-45-67
• Саше — 200 ₽
  Сбербанк: +7 999 987-65-43
```

**Для выигравших:**
```
🎰 Игра #42 завершена!

🎉 Ты выиграл 800 ₽!

Тебе должны перевести:
• Вася — 800 ₽
```

**Acceptance criteria:**
- Номера телефонов форматируются через HTML-тег `<code>` для копирования по клику в Telegram.
- Победителям **не показываются** телефоны должников — только имена.
- Если сумма перевода = 0 (игрок вышел в ноль), бот отправляет: «🤝 Ты остался при своих. Никому ничего не должен».

### 3.8 Итоговая статистика в общий чат

**Описание:** публичная сводка результатов, публикуемая в групповом чате после успешного расчёта.

**Формат:**
```
🎰 Игра #42 завершена
Длительность: 3ч 20мин
Банк: 15 000 ₽
Игроков: 6

📊 Результаты:
🥇 Коля: +3200 ₽ (бай-ин + 2 докупа)
🥈 Саша: +1500 ₽ (бай-ин + 0 докупов)
🥉 Миша: +300 ₽ (бай-ин + 1 докуп)
❌ Петя: −500 ₽ (бай-ин + 1 докуп)
❌ Дима: −1500 ₽ (бай-ин + 2 докупа)
❌ Вася: −3000 ₽ (бай-ин + 3 докупа)

💸 Переводы:
• Вася → Коля: 3000 ₽
• Дима → Коля: 200 ₽
• Дима → Саша: 1300 ₽
• Петя → Миша: 500 ₽
```

**Acceptance criteria:**
- Длительность рассчитывается как разница между `created_at` и `finished_at`.
- Игроки сортируются по результату (от максимального выигрыша к максимальному проигрышу).
- Медали 🥇🥈🥉 — только для игроков с положительным результатом, первым трём.

## 4. Рекомендации по техническому стеку

### 4.1 Язык и библиотеки
- **Язык:** Go 1.22+
- **Telegram Bot Framework:** [`github.com/go-telegram/bot`](https://github.com/go-telegram/bot) v1.18+
  - Zero-dependencies, поддерживает Bot API 9.3+, активно развивается.
  - Документация: <https://github.com/go-telegram/bot>
- **UI-компоненты (опционально):** [`github.com/go-telegram/ui`](https://github.com/go-telegram/ui) — готовые inline-клавиатуры и другие элементы.
- **База данных:** SQLite через [`modernc.org/sqlite`](https://pkg.go.dev/modernc.org/sqlite) — **pure Go** драйвер, без CGO. Это гарантирует, что бинарник собирается статически и легко переносится между машинами.
- **Логирование:** стандартный `log/slog` с выводом в файл + ротация через [`gopkg.in/natefinch/lumberjack.v2`](https://github.com/natefinch/lumberjack).

### 4.2 Архитектура получения апдейтов
- **Long polling** — бот инициирует запросы к Telegram API, не требует публичного IP, HTTPS или открытых портов.
- Идеально подходит для домашнего сервера без DNS/SSL-конфигурации.

### 4.3 Хостинг
- **Целевой хостинг:** личный сервер друга.
- **Деплой:** ручная сборка бинарника (`go build`) и копирование на сервер, запуск вручную.
- **Рекомендация:** завернуть в `systemd unit` или `tmux/screen` сессию для автоматического перезапуска при сбое (опционально, не в MVP).

### 4.4 Конфигурация
- Все секреты и параметры — через переменные окружения или `.env` файл:
  - `TELEGRAM_BOT_TOKEN` — токен бота от @BotFather.
  - `DB_PATH` — путь к файлу SQLite (по умолчанию `./poker.db`).
  - `LOG_PATH` — путь к файлу логов (по умолчанию `./bot.log`).
  - `ALLOWED_CHAT_ID` — ID группового чата друзей (ограничение доступа).
  - `ADMIN_USER_IDS` — список Telegram ID владельцев бота (для отладочных команд).

### 4.5 Ограничение доступа
Поскольку это бот «только для своих», доступ должен быть ограничен:
- Бот реагирует только на сообщения из `ALLOWED_CHAT_ID` и личных чатов пользователей, состоящих в этом чате.
- Неизвестным пользователям бот отвечает: «Этот бот работает только в закрытой группе. Обратись к владельцу».
- Проверка членства в группе через `getChatMember` при первом взаимодействии, кэширование результата.

## 5. Концептуальная модель данных

### 5.1 Таблица `players`
Зарегистрированные игроки.

| Поле              | Тип       | Описание                                      |
|-------------------|-----------|-----------------------------------------------|
| `telegram_id`     | INTEGER   | PRIMARY KEY, Telegram user ID                 |
| `telegram_username` | TEXT    | `@username` (может быть NULL)                 |
| `display_name`    | TEXT      | Отображаемое имя (по умолчанию из профиля)    |
| `phone_number`    | TEXT      | Номер в формате `+7XXXXXXXXXX`                |
| `bank_name`       | TEXT      | Название банка                                |
| `created_at`      | TIMESTAMP | Время регистрации                             |
| `updated_at`      | TIMESTAMP | Время последнего обновления                   |

### 5.2 Таблица `games`
Игровые сессии.

| Поле           | Тип       | Описание                                                   |
|----------------|-----------|------------------------------------------------------------|
| `id`           | INTEGER   | PRIMARY KEY, AUTOINCREMENT                                 |
| `chat_id`      | INTEGER   | ID группового чата, где создана игра                       |
| `creator_id`   | INTEGER   | FK → `players.telegram_id`                                 |
| `buy_in`       | INTEGER   | Размер бай-ина в рублях                                    |
| `hub_message_id` | INTEGER | ID сообщения-хаба в групповом чате (для `editMessageText`) |
| `status`       | TEXT      | `active` / `collecting_results` / `finished`               |
| `created_at`   | TIMESTAMP | Время создания                                             |
| `finished_at`  | TIMESTAMP | Время финального расчёта (NULL пока не завершена)          |

### 5.3 Таблица `game_participants`
Участники конкретной игры.

| Поле              | Тип       | Описание                                                  |
|-------------------|-----------|-----------------------------------------------------------|
| `id`              | INTEGER   | PRIMARY KEY, AUTOINCREMENT                                |
| `game_id`         | INTEGER   | FK → `games.id`                                           |
| `player_id`       | INTEGER   | FK → `players.telegram_id`                                |
| `rebuy_count`     | INTEGER   | Количество докупов (>= 0)                                 |
| `final_chips`     | INTEGER   | Остаток фишек/рублей в конце (NULL пока не заполнено)     |
| `results_confirmed` | BOOLEAN | Игрок подтвердил свои данные                              |
| `joined_at`       | TIMESTAMP | Время присоединения к игре                                |

**UNIQUE constraint:** `(game_id, player_id)` — один игрок может быть только раз в одной игре.

### 5.4 Таблица `settlements`
Рассчитанные переводы (для истории и отладки).

| Поле         | Тип     | Описание                        |
|--------------|---------|---------------------------------|
| `id`         | INTEGER | PRIMARY KEY, AUTOINCREMENT      |
| `game_id`    | INTEGER | FK → `games.id`                 |
| `from_player_id` | INTEGER | FK → `players.telegram_id` |
| `to_player_id`   | INTEGER | FK → `players.telegram_id` |
| `amount`     | INTEGER | Сумма перевода в рублях         |

### 5.5 Технические детали работы с БД
- **WAL-режим** (`PRAGMA journal_mode=WAL`) — обязательно включить при инициализации, чтобы корректно обрабатывать конкурентные нажатия кнопок несколькими игроками одновременно.
- **Foreign keys** — включить через `PRAGMA foreign_keys=ON`.
- **Миграции** — простой подход: при старте бота выполняются `CREATE TABLE IF NOT EXISTS ...` для всех таблиц. При будущих изменениях схемы — ручные ALTER-ы, либо миграция на `goose` (в Future Enhancements).

## 6. Принципы UI-дизайна

### 6.1 Общие принципы
- **Минимализм:** каждое сообщение содержит только необходимую информацию.
- **Идемпотентность:** повторное нажатие той же кнопки не должно ломать состояние (например, повторное «Присоединиться» даёт уведомление «Ты уже в игре»).
- **Визуальная иерархия:** эмодзи в начале строк для быстрого сканирования (🎰 игра, 💰 банк, 💸 перевод, ✅⏳ статусы).
- **Двуязычность:** MVP — только русский язык. Поддержка английского — в Future Enhancements.

### 6.2 Паттерны сообщений
- **Сообщения в группе** — максимально информативные, служат «доской объявлений».
- **Сообщения в личке** — интерактивные, для ввода данных.
- **Уведомления (callback alerts)** — короткие подтверждения действий («Ты докупился!»).

### 6.3 Форматирование
- Использовать **HTML parse mode** (не MarkdownV2) — меньше экранирования, удобнее работать с номерами телефонов.
- Номера телефонов в финальных сообщениях оборачивать в `<code>+7 999 123-45-67</code>` для копирования по клику.
- Жирный шрифт `<b>` — только для ключевых чисел (суммы, результаты).

## 7. Технические соображения для сложных функций

### 7.1 Алгоритм минимизации транзакций

**Цель:** свести итоговые долги к минимальному количеству переводов.

**Входные данные:** массив `balances[i]` — результат каждого игрока (положительный = выиграл, отрицательный = проиграл), сумма = 0.

**Жадный алгоритм (рекомендуемый для MVP):**
```
1. Разделить игроков на debtors (balance < 0) и creditors (balance > 0).
2. Отсортировать обе группы по абсолютному значению (по убыванию).
3. Пока есть и должники, и кредиторы:
   a. Взять первого должника D и первого кредитора C.
   b. amount = min(|D.balance|, C.balance)
   c. Создать перевод: D → C на сумму amount.
   d. D.balance += amount; C.balance -= amount.
   e. Если balance у кого-то стал 0 — убрать из списка.
4. Вернуть список переводов.
```

**Сложность:** O(n log n) на сортировку + O(n) на обход. Для 10 игроков — мгновенно.

**Оптимальность:** жадный алгоритм даёт ≤ (n−1) переводов, что на практике близко к оптимуму. Истинно минимальное решение — NP-hard задача (partition problem), но для 4–10 игроков разница незаметна.

### 7.2 Обработка конкурентных нажатий кнопок
- Все операции, изменяющие состояние игры (присоединение, докуп, подтверждение результата), должны выполняться внутри **SQLite-транзакции**.
- WAL-режим обеспечивает, что чтения не блокируют записи.
- При конфликте (например, два одновременных нажатия «Завершить игру») побеждает первая транзакция, вторая получает `SQLITE_BUSY` → бот показывает пользователю «Попробуй ещё раз».

### 7.3 Graceful shutdown
- При получении SIGINT/SIGTERM бот должен:
  1. Остановить получение новых апдейтов (`cancel` контекста).
  2. Дождаться завершения обработки текущих хендлеров (с таймаутом ~5 секунд).
  3. Закрыть подключение к SQLite (`db.Close()`).
  4. Закрыть файл логов.
- Реализация — через `signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)`.

### 7.4 Логирование
- **Библиотека:** `log/slog` (стандартная, structured logging) + `lumberjack.v2` для ротации.
- **Куда пишем:** файл `./bot.log`, ротация при достижении 10 MB, хранение 5 бэкапов.
- **Что логируем:**
  - `INFO`: старт/стоп бота, создание игры, завершение игры, финальные расчёты.
  - `WARN`: расхождение банка, попытки неавторизованных действий.
  - `ERROR`: ошибки API Telegram, ошибки БД, паники в хендлерах.
- **Что НЕ логируем:** номера телефонов, полные имена (для минимизации рисков при утечке логов).

### 7.5 Обновление сообщения-хаба
- Каждое действие (присоединение, докуп, отмена докупа, подтверждение результата) триггерит `editMessageText` для сообщения-хаба.
- **Rate limit Telegram:** редактирование того же сообщения чаще 1 раза в секунду может привести к ошибкам. Решение: **дебаунсинг** — если за последнюю секунду уже было обновление, отложить следующее через `time.AfterFunc`.
- В MVP можно обойтись без дебаунсинга и просто обрабатывать ошибку `429 Too Many Requests`, повторяя с задержкой.

## 8. Соображения по безопасности

### 8.1 Защита от несанкционированного доступа
- **Whitelist по чату:** бот реагирует только на сообщения из чата с `ALLOWED_CHAT_ID` и личных чатов участников этого чата.
- **Проверка членства** через `getChatMember` при первом обращении пользователя.

### 8.2 Защита персональных данных
- Номера телефонов и названия банков хранятся в plain text в SQLite — шифрование избыточно для локальной БД на личном сервере.
- **Бэкапы БД:** владелец бота вручную делает копию `poker.db` периодически (не в MVP).
- Логи не содержат PII (см. 7.4).

### 8.3 Защита от мошенничества внутри группы
Поскольку бот для доверенной группы друзей, этот вектор угроз **не является приоритетом**. Тем не менее:
- Любой участник может изменять **только свои** данные (проверка `callback_query.from.id`).
- Все операции логируются, при подозрении на манипуляции можно поднять логи.

### 8.4 Защита токена бота
- `TELEGRAM_BOT_TOKEN` хранится в переменной окружения или `.env` файле, **не коммитится в git**.
- `.env` добавляется в `.gitignore`.

## 10. Потенциальные проблемы и способы их решения

| Проблема | Вероятность | Решение |
|----------|-------------|---------|
| Игроки забывают нажать «Докупиться» во время игры | Высокая | Возможность скорректировать количество докупов на финальном этапе (уже в MVP) |
| Расхождение банка при подсчёте | Средняя | Блокирующая валидация — расчёт не запускается до устранения расхождения |
| Несколько одновременных нажатий кнопок | Средняя | SQLite WAL-режим + транзакции, идемпотентные хендлеры |
| Telegram rate limits при частом редактировании сообщения-хаба | Низкая | Обработка `429 Too Many Requests` с retry, опционально — дебаунсинг |
| Игрок случайно нажал «Завершить игру» | Средняя | Подтверждение через `callback_query` alert |
| Потеря данных при падении сервера | Низкая | WAL-режим обеспечивает durability, ручные бэкапы файла БД |
| Бот недоступен в момент игры | Средняя | Systemd автоперезапуск (опционально), логирование ошибок для постмортема |
| Новый участник компании не может сразу играть | Низкая | Онбординг занимает 30 секунд, делается заранее |

## 12. Глоссарий

- **Бай-ин** — стартовая сумма, которую игрок вносит для участия в игре (по умолчанию 1000 ₽).
- **Докуп (rebuy)** — дополнительная покупка фишек на сумму бай-ина во время игры.
- **Банк** — общая сумма денег всех игроков в текущей игре = `Σ(бай-ин + докупы)`.
- **СБП** — Система Быстрых Платежей, позволяющая переводить деньги по номеру телефона между банками РФ.
- **Сообщение-хаб** — обновляемое сообщение в групповом чате, отражающее текущее состояние игры.
- **WAL** — Write-Ahead Logging, режим работы SQLite с улучшенной поддержкой конкурентного доступа.
